Transactions in SQL

Transactions in SQL

A Transaction in SQL is a sequence of operations performed as a single logical unit of work. A transaction must be completed in its entirety or not executed at all. Transactions ensure data integrity and consistency, adhering to the ACID properties (Atomicity, Consistency, Isolation, Durability).

ACID Properties

Transactions in SQL are governed by the following ACID properties:

  • Atomicity: Ensures that all operations within a transaction are completed successfully. If any operation fails, the transaction is rolled back, and none of the operations are applied.
  • Consistency: Ensures that a transaction brings the database from one valid state to another, maintaining database rules and constraints.
  • Isolation: Ensures that transactions are executed in isolation from one another, so concurrent transactions do not interfere with each other.
  • Durability: Ensures that once a transaction is committed, its changes are permanent and will survive system failures.

Basic Transaction Commands

The primary commands used to manage transactions are:

BEGIN TRANSACTION

The BEGIN TRANSACTION statement starts a new transaction:

BEGIN TRANSACTION;

COMMIT

The COMMIT statement saves all changes made during the transaction. This makes the changes permanent:

COMMIT;

ROLLBACK

The ROLLBACK statement undoes all changes made during the transaction if an error occurs or if the transaction needs to be aborted:

ROLLBACK;

Example of a Transaction

Here is a practical example of a transaction that transfers money between two accounts:

BEGIN TRANSACTION;

    -- Deduct amount from Account A
    UPDATE Accounts
    SET Balance = Balance - 100
    WHERE AccountID = 'A';

    -- Add amount to Account B
    UPDATE Accounts
    SET Balance = Balance + 100
    WHERE AccountID = 'B';

    -- Check if both operations are successful
    IF @@ERROR != 0
    BEGIN
        ROLLBACK;
        PRINT 'Transaction failed and rolled back.';
    END
    ELSE
    BEGIN
        COMMIT;
        PRINT 'Transaction succeeded and committed.';
    END

Transaction Isolation Levels

Isolation levels determine how transaction changes are visible to other transactions. Common isolation levels include:

  • Read Uncommitted: Allows transactions to read data that has been modified but not yet committed by other transactions.
  • Read Committed: Ensures that transactions can only read committed data. It prevents dirty reads but may allow non-repeatable reads.
  • Repeatable Read: Ensures that if a transaction reads a row, it will see the same data if it reads the row again during the same transaction. It prevents non-repeatable reads but may allow phantom reads.
  • Serializable: Ensures complete isolation from other transactions by executing transactions sequentially. It prevents dirty reads, non-repeatable reads, and phantom reads.

Managing Transactions in Different SQL Dialects

SQL Server

In SQL Server, use BEGIN TRANSACTION, COMMIT, and ROLLBACK for transaction management:

BEGIN TRANSACTION;

    -- Transaction logic

COMMIT;

MySQL

In MySQL, transactions are managed using similar commands:

START TRANSACTION;

    -- Transaction logic

COMMIT;

Oracle

In Oracle, transactions are handled automatically, but you can use COMMIT and ROLLBACK for explicit transaction control:

BEGIN;

    -- Transaction logic

COMMIT;
