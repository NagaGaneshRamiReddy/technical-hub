Salesforce Security Model

What is the Salesforce Security Model ?

The Salesforce Security Model is designed to protect data and ensure that users only have access to the information they need. It encompasses various layers of security, including user authentication, data access controls, and platform security features to safeguard your Salesforce environment.

Key Components of the Salesforce Security Model

  • User Authentication: Methods used to verify user identities, including username and password, multi-factor authentication (MFA), and single sign-on (SSO).
  • Profile and Permission Sets: Controls what users can see and do in Salesforce. Profiles define baseline permissions, while permission sets provide additional access on top of profiles.
  • Role Hierarchy: Defines the organizational structure in Salesforce, allowing access to data based on a user’s position in the hierarchy.
  • Sharing Rules: Determine how data is shared among users, including public and private access levels, and rules for sharing records.
  • Field-Level Security: Controls access to specific fields within objects, ensuring sensitive data is protected.
  • Object-Level Security: Manages access to different objects (e.g., Accounts, Opportunities) and their associated records.
  • Login IP Ranges and Hours: Restrict access to Salesforce based on IP addresses and login hours to enhance security.
  • Audit Trails: Tracks changes and access to data, helping to monitor and review user activity and system changes.

User Authentication

  • Username and Password: Basic authentication method where users log in with a username and password.
  • Multi-Factor Authentication (MFA): Adds an extra layer of security by requiring additional verification (e.g., a code sent to a mobile device).
  • Single Sign-On (SSO): Allows users to access Salesforce with credentials from an external identity provider, improving user experience and security.

Profile and Permission Sets

  • Profiles: Define a user’s permissions and access rights across the Salesforce platform. Each user is assigned one profile.
  • Permission Sets: Provide additional permissions beyond those granted by the user’s profile, allowing more granular access control.

Role Hierarchy

  • Role Setup: Organize users into a hierarchy based on their role within the organization.
  • Data Access: Users at higher levels in the hierarchy can access the data owned by users below them, facilitating data visibility and reporting.

Sharing Rules

  • Public Access: Configures objects to be accessible by all users or specific groups.
  • Private Access: Restricts access to records so that only the owner and users above them in the role hierarchy can view or edit the records.
  • Rule-Based Sharing: Defines automatic sharing rules to provide access based on criteria such as record owner or field values.

Field-Level Security

  • Field Visibility: Control which fields are visible or read-only based on user profiles and permission sets.
  • Field-Level Security Settings: Adjust settings to protect sensitive data and ensure compliance with privacy regulations.

Object-Level Security

  • Object Permissions: Manage access to objects through profiles and permission sets, determining what actions users can perform (e.g., read, create, edit, delete).
  • Record Types: Use different record types to manage access and functionality within the same object, tailoring experiences to different user groups.

Login IP Ranges and Hours

  • IP Restrictions: Define which IP addresses can access Salesforce, adding an extra layer of security against unauthorized access.
  • Login Hours: Restrict login access to specific hours to prevent unauthorized access during off-hours.

Audit Trails

  • Setup Audit Trail: Track changes made in Salesforce Setup, including configuration changes and administrative actions.
  • Field History Tracking: Monitor changes to specific fields and review historical data for compliance and auditing purposes.

Example Use Case: Securing Customer Data

  • Profile Management: Assign profiles to users based on their roles to ensure appropriate access to customer data.
  • Field-Level Security: Restrict access to sensitive fields such as customer credit card information.
  • Sharing Rules: Create sharing rules to allow only specific teams to view and edit customer records based on their role and region.
  • Audit Trails: Monitor changes to customer data and access logs to ensure compliance with data protection regulations.

Summary

The Salesforce Security Model provides a comprehensive framework to protect data and control user access within the platform. By implementing robust authentication methods, defining profiles and permissions, setting up role hierarchies, and utilizing sharing rules, organizations can ensure their Salesforce environment is secure and compliant with industry standards.
