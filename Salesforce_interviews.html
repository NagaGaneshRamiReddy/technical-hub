Top Salesforce Interview Questions

  1. 1. What is Salesforce and what are its main products?
    Salesforce is a cloud-based CRM (Customer Relationship Management) platform designed to help businesses manage customer relationships, streamline processes, and increase profitability. The main Salesforce products include:
    • Sales Cloud: For sales process management, from lead tracking to closing deals.
    • Service Cloud: For managing customer support and case resolution.
    • Marketing Cloud: For personalized marketing and customer journeys.
    • Commerce Cloud: Supports e-commerce activities.
    • Community Cloud: Allows businesses to build branded communities.
    • AppExchange: A marketplace for third-party applications that integrate with Salesforce.
    • Platform/Force.com: A PaaS for creating custom apps.
  2. 2. Explain the difference between Salesforce.com and Force.com.
    Salesforce.com refers to Salesforce’s CRM product suite, which includes Sales, Service, Marketing, and other Clouds. Force.com is the underlying platform (PaaS) that allows developers to build and deploy custom applications that work within the Salesforce ecosystem, focusing on development tools and services.
  3. 3. What is Salesforce CRM, and how does it help businesses?
    Salesforce CRM is a tool that helps businesses manage interactions with current and potential customers. It enables companies to improve customer service, personalize marketing, close sales, and monitor performance through organized, data-driven workflows and insights.
  4. 4. Describe the multi-tenant architecture of Salesforce.
    Salesforce’s multi-tenant architecture allows multiple organizations (tenants) to share the same infrastructure and resources while securely separating each organization's data. This setup allows for cost efficiency, automatic upgrades, and consistent performance for all users.
  5. 5. Explain the difference between Salesforce Classic and Lightning.
    Salesforce Classic is the older user interface of Salesforce, while Salesforce Lightning is the modernized UI with a more intuitive design and optimized user experience. Lightning also brings advanced development frameworks, allowing for more dynamic, interactive applications.
  6. 6. What is an App in Salesforce?
    In Salesforce, an App is a collection of items, such as tabs, reports, and dashboards, organized to provide a specific set of functionalities. Apps can be standard (built-in) or custom (tailored to business requirements) and enable streamlined access to specific objects and records.
  7. 7. What are Standard and Custom Objects in Salesforce?
    Standard Objects are built-in Salesforce objects like Account, Contact, and Opportunity. Custom Objects are user-created objects designed to store data unique to a business, enabling custom functionality and specific workflows that standard objects may not support.
  8. 8. Explain the difference between a Lookup and Master-Detail relationship.
    A Lookup relationship links two objects, allowing related data but maintaining separate ownership and security. In contrast, a Master-Detail relationship links two objects in a tightly bound connection, with the child record’s lifecycle and permissions dependent on the parent record.
  9. 9. What are Record Types in Salesforce, and how are they used?
    Record Types allow different business processes, picklist values, and page layouts for the same object, depending on the user role or process type. For example, Sales and Support teams can use different record types to see layouts relevant to their tasks.
  10. 10. How does the Salesforce security model work (Profiles,
    Permission Sets, Sharing Rules)?
    Salesforce’s security model includes various layers to control access:
    • Profiles: Define baseline permissions (read, edit, delete) and field-level security for users.
    • Permission Sets: Allow additional permissions on top of profiles.
    • Sharing Rules: Grant record-level access based on record ownership or criteria.
  11. 11. What is a Schema Builder in Salesforce?
    The Schema Builder is a graphical tool within Salesforce that provides a visual representation of objects and their relationships. It allows for the creation and modification of custom fields and relationships, facilitating easy, no-code schema customization.
  12. 12. How do different types of fields work in Salesforce (Formula, Roll-Up Summary, etc.)?
    Different field types serve various purposes:
    • Formula Fields: Automatically calculate values based on other fields.
    • Roll-Up Summary Fields: Aggregate child records’ data, such as summing amounts on related records.
    • Picklists, Checkbox, Currency: Allow standardized values for fields, improving data accuracy and reporting.
  13. 13. What are Relationships in Salesforce, and how many types are there?
    Relationships link Salesforce objects to one another, defining how data is interconnected. Main types include:
    • Lookup Relationship: Loose association, independent of the related record’s lifecycle.
    • Master-Detail Relationship: Dependent relationship with the child record tied to the parent.
    • Junction Object (Many-to-Many): Created using two master-detail relationships to link two objects.
  14. 14. Describe the difference between External ID and Unique ID.
    An External ID is a field marked to store an identifier from an external system for easy data import. Unique IDs ensure that each record in Salesforce has a distinct identifier, which helps avoid duplicate records.
  15. 15. How can you use the Data Import Wizard vs. Data Loader in Salesforce?
    Data Import Wizard is an in-browser tool for importing smaller data sets and supports only standard and custom objects. Data Loader is a more robust, standalone application that handles large data volumes, supports additional objects, and offers more options for data operations (like export and delete).
  16. 16. What are Junction Objects, and how are they used?
    Junction Objects enable many-to-many relationships between two objects. They are created with two master-detail relationships on a third object, such as connecting Students and Courses with a Course Enrollment junction object.
  17. 17. Explain the difference between an Object and a Record in Salesforce.
    An Object is a database table that stores data about a specific type of item (e.g., Account or Contact). A Record is a single instance of that object, such as a particular customer in the Account object.
  18. 18. What is a Hierarchical Relationship?
    A Hierarchical Relationship is a unique type of lookup relationship available only on the User object. It allows users to define a reporting structure within an organization.
  19. 19. How do you create Many-to-Many relationships in Salesforce?
    Many-to-Many relationships are created using a Junction Object. This object has two master-detail relationships linking the two objects you want to connect, allowing records from each object to be associated with multiple records from the other.
  20. 20. How do you track field history in Salesforce?
    Field history tracking can be enabled on specific fields, allowing Salesforce to automatically log changes in a history related list on records. You can track up to 20 fields per object for standard objects and 50 for custom objects.
  21. 21. What is OWD (Organization-Wide Default) and how is it configured?
    OWD (Organization-Wide Default) sets the baseline record-level access for each object within an organization, establishing the default level of access users have to records they don’t own. OWDs can be set to Private, Public Read Only, Public Read/Write, or Controlled by Parent, depending on the needs. OWDs are configured in the Sharing Settings under Setup.
  22. 22. What is Role Hierarchy, and how does it work?
    Role Hierarchy in Salesforce is a data access model that allows users to have access to records owned by people below them in the hierarchy. It allows for record visibility across an organization, ensuring that users in higher roles can view or manage records owned by lower roles.
  23. 23. Explain the difference between Profiles and Roles.
    Profiles control a user’s permissions, such as which apps, tabs, and objects they can access and what actions they can perform on records (read, edit, delete). Roles, on the other hand, define a user’s place in the Role Hierarchy and primarily influence record access, determining which records users can view or edit based on data ownership.
  24. 24. What are Permission Sets, and when would you use them?
    Permission Sets extend user permissions without altering their Profile. They’re useful for granting specific permissions to individual users or user groups who need additional access to objects, fields, or applications without creating new profiles.
  25. 25. How do you restrict data access using Sharing Rules?
    Sharing Rules provide additional access to records for users or groups beyond what’s defined by OWD. They grant access based on record ownership or specific criteria (e.g., all records where Account Status = “Active”). This is particularly useful for collaborative tasks across different departments.
  26. 26. What is a Public Group?
    A Public Group is a collection of users, roles, or other public groups. It’s used in sharing rules, folder access, and approval processes to provide access to data or functionality for a specified group of people.
  27. 27. Explain the purpose of the “View All” and “Modify All” permissions.
    “View All” and “Modify All” are permissions that override standard sharing settings, giving users access to view or edit all records within a specific object. This is typically used for system administrators or for those who need full access for specific business processes.
  28. 28. What is Field-Level Security, and how is it configured?
    Field-Level Security restricts user access to specific fields within an object, enabling administrators to hide sensitive information. It’s configured within Profiles and Permission Sets by setting fields as visible or read-only based on user role.
  29. 29. How would you use Login IP Ranges and Login Hours?
    Login IP Ranges restrict access to Salesforce from specified IP addresses, while Login Hours limit access during certain hours. These features are configured in Profiles to enhance security by ensuring users can only log in from trusted networks and during permitted times.
  30. 30. What are Apex Sharing Rules?
    Apex Sharing Rules allow developers to create programmatic sharing settings for custom business logic. They are used in Apex code to explicitly define sharing rules for records, offering a customized approach beyond standard Sharing Rules.
  31. 31. What are Workflow Rules, and what actions can they perform?
    Workflow Rules automate standard internal procedures in Salesforce, triggering actions like email alerts, field updates, tasks, or outbound messages. They’re set up with criteria-based logic to execute these actions automatically when conditions are met.
  32. 32. Describe the differences between Workflow, Process Builder, and Flow.
    Workflow is limited to four types of actions (alerts, tasks, field updates, and messages), whereas Process Builder is more powerful, handling multiple criteria, actions, and sequences in a single process. Flow is the most advanced, enabling complex, multi-step workflows and screens for user interaction.
  33. 33. How does the Process Builder differ from Workflow?
    Process Builder offers more flexibility, allowing multiple conditions and actions in a single process, whereas Workflow is limited to simpler, one-condition/one-action automation.
  34. 34. What are Approval Processes in Salesforce?
    Approval Processes are multi-step workflows that automatically route records to designated approvers based on criteria. They are often used for request-based business processes, like budget or time-off requests.
  35. 35. Explain the concept of Time-Dependent Workflow Actions.
    Time-Dependent Workflow Actions execute automatically after a set time delay. For example, an email reminder could be sent 7 days before an event, based on specified conditions.
  36. 36. What is an Escalation Rule?
    Escalation Rules automatically reassign or escalate cases when they meet certain criteria (e.g., unresolved cases after 24 hours). They’re essential in maintaining service-level agreements (SLAs).
  37. 37. How would you create an Automated Approval Process?
    Automated Approval Processes involve defining entry criteria, assigning approval steps, specifying actions (approve/reject), and establishing conditions for notifications. This can be set up in Setup > Approval Processes.
  38. 38. What are Quick Actions, and how are they used?
    Quick Actions allow users to perform tasks quickly, like creating a record, sending an email, or updating fields. They are added to pages or layouts to streamline routine actions.
  39. 39. How does Lightning Flow work, and what are its main use cases?
    Lightning Flow provides automation with Flow Builder, allowing users to create complex, multi-step processes. Use cases include guided user experiences, handling multiple conditions, and performing advanced data manipulation.
  40. 40. What is Einstein Next Best Action?
    Einstein Next Best Action provides AI-driven recommendations for users, suggesting the best action based on predictive models, historical data, and business rules, which helps in personalized customer interactions.
  41. 41. What types of reports can be created in Salesforce?
    Salesforce supports Tabular, Summary, Matrix, and Joined Reports, each tailored to different analysis types (e.g., simple lists, grouped data, pivot tables, and complex report combinations).
  42. 42. How do Report Types work in Salesforce?
    Report Types define the data relationships available for reports. Standard Report Types are provided by Salesforce, while Custom Report Types allow for more granular control of related data inclusion.
  43. 43. What are Custom Report Types, and how do they differ from Standard Report Types?
    Custom Report Types let users specify object relationships and choose which fields appear in reports, offering more customization compared to predefined Standard Report Types.
  44. 44. Explain how dashboards work in Salesforce.
    Dashboards visualize report data in components (like charts and tables). They offer real-time insights into metrics and KPIs, showing a comprehensive view of business performance.
  45. 45. How would you create a Dynamic Dashboard?
    A Dynamic Dashboard shows data based on the viewing user’s access, created by enabling “View Dashboard As” settings in Dashboard Properties.
  46. 46. What are Report Filters and how are they used?
    Report Filters narrow down report data based on specific criteria, allowing users to focus on the most relevant records.
  47. 47. How does Conditional Highlighting work in reports?
    Conditional Highlighting visually emphasizes key metrics in Summary and Matrix reports, like highlighting high sales figures, helping in quick data analysis.
  48. 48. Explain the purpose of Joined Reports.
    Joined Reports allow combining multiple reports into a single view, useful for comparing different data sets side-by-side.
  49. 49. What is the difference between Summary and Matrix reports?
    Summary Reports group data by rows, while Matrix Reports group data by both rows and columns, offering a pivot table layout.
  50. 50. How would you set up a Reporting Snapshot?
    A Reporting Snapshot captures and stores report data at scheduled intervals. It’s set up in Setup > Reporting Snapshots to monitor changes over time.
  51. 51. What is Apex in Salesforce?
    Apex is Salesforce’s proprietary programming language similar to Java. It’s used to add business logic, handle complex automation, and interact with Salesforce’s database through server-side programming.
  52. 52. What are Apex Triggers, and how are they used?
    Apex Triggers are pieces of code that execute before or after specific events on a Salesforce record, like insert, update, delete, or undelete. Triggers are often used for tasks like validating data, logging changes, or initiating other automated actions.
  53. 53. Explain the concept of Trigger Context Variables.
    Trigger Context Variables are variables in Apex that store data about the trigger event, such as Trigger.new (new records), Trigger.old (old records), Trigger.isInsert, and Trigger.isUpdate. These help developers know which records are affected and the nature of the change.
  54. 54. What are Governor Limits in Salesforce?
    Governor Limits are Salesforce’s way of ensuring efficient resource usage in a shared environment. They impose limits on the number of DML operations, queries, CPU time, etc., to prevent one user from overconsuming resources.
  55. 55. Describe the difference between a Trigger and a Workflow.
    A Trigger allows for more complex, custom logic and can execute before or after record changes, while Workflow offers simple, predefined actions like field updates or email alerts triggered by record criteria.
  56. 56. Explain the purpose of the @future annotation in Apex.
    The @future annotation in Apex is used to run methods asynchronously, allowing time-consuming tasks like callouts to external systems or batch processing to be performed without delaying the main execution.
  57. 57. What is Batch Apex and when is it used?
    Batch Apex is used to process large data sets asynchronously by breaking them into smaller chunks, which helps bypass governor limits and efficiently handle thousands of records at once. It's ideal for large data processing.
  58. 58. Explain the difference between Synchronous and Asynchronous Apex.
    Synchronous Apex processes tasks immediately, but it’s limited by governor limits, while Asynchronous Apex (using @future, Batch Apex, Queueable Apex) can handle more resource-intensive tasks in the background.
  59. 59. What is a Visualforce page, and how is it used?
    Visualforce is a markup language in Salesforce for creating custom user interfaces. It’s used for building pages that interact with the backend and are more customizable than standard pages, especially in classic Salesforce.
  60. 60. Describe the purpose of Custom Controllers and Controller Extensions.
    Custom Controllers are Apex classes associated with Visualforce pages, managing the page’s behavior. Controller Extensions enhance existing controllers by adding additional functionality or overriding specific methods.
  61. 61. What is Salesforce Lightning?
    Salesforce Lightning is a component-based framework for developing dynamic web applications for mobile and desktop. It includes the Lightning Experience UI and Lightning components for building modern applications.
  62. 61. What is Salesforce Lightning?
    Salesforce Lightning is a component-based framework for developing dynamic web applications for mobile and desktop. It includes the Lightning Experience UI and Lightning components for building modern applications.
  63. 62. Explain the difference between Aura and Lightning Web Components (LWC).
    Aura Components are Salesforce’s original component model, using proprietary code, while Lightning Web Components (LWC) are based on web standards, allowing faster, lighter, and more efficient development using standard JavaScript.
  64. 63. What is the Lightning App Builder?
    Lightning App Builder is a point-and-click tool for creating customized Lightning pages using components. It’s useful for designing home pages, record pages, and app pages tailored to specific business needs.
  65. 64. How does the Lightning Design System (SLDS) work?
    SLDS provides a library of styling guidelines, components, and resources that ensure consistency across the Salesforce UI, helping developers and admins build interfaces that look and feel like native Salesforce elements.
  66. 65. Explain Lightning Data Service (LDS).
    LDS is a service that provides seamless access to Salesforce data in Lightning components without the need for Apex code. It handles caching and data refresh, making it efficient for CRUD operations on Salesforce records.
  67. 66. How would you implement navigation between components in LWC?
    Navigation in LWC is implemented using the lightning/navigation module, which allows developers to navigate to records, pages, or external links programmatically within the Lightning Experience.
  68. 67. What is an Event in Lightning, and how is it handled?
    Events in Lightning are mechanisms for communication between components. In Aura, events are categorized as application or component events, while in LWC, standard DOM events are used, which can be dispatched and listened to for inter-component communication.
  69. 68. Explain the difference between Lightning Record Page and App Page.
    A Lightning Record Page is designed for viewing and editing records of a specific object, whereas an App Page serves as a homepage or dashboard for the entire app, often aggregating information from multiple objects.
  70. 69. What are @track, @api, and @wire decorators in LWC?
    @track makes fields reactive, updating the UI when they change; @api exposes properties and methods for parent-to-child communication; and @wire connects components to Salesforce data for automatic data refresh.
  71. 70. How would you debug an LWC component?
    Debugging in LWC involves using browser developer tools, console.log statements, and the Lightning Debug Mode in Salesforce for better error tracing and faster load times for JavaScript changes.
  72. 71. What is the purpose of REST and SOAP APIs in Salesforce?
    REST API and SOAP API allow for integrating Salesforce with other systems by enabling data exchange. REST API is lightweight and ideal for mobile and web applications, while SOAP API offers a more robust, structured approach, suitable for enterprise applications.
  73. 72. How does OAuth 2.0 work in Salesforce?
    OAuth 2.0 is an authorization protocol allowing external applications to securely access Salesforce data on behalf of a user without sharing passwords. It’s widely used for granting permissions via tokens.
  74. 73. Explain the concept of External Services in Salesforce.
    External Services simplify integrations by connecting to external REST-based services. It allows Salesforce to call external APIs using declarative tools by importing service definitions directly.
  75. 74. What is a Connected App, and why is it used?
    A Connected App is a framework that lets an external application connect with Salesforce through APIs. It enables secure integration, managing permissions, and providing user authentication.
  76. 75. How does the Salesforce API Limit work?
    Salesforce API Limits govern the number of API calls allowed within a 24-hour period. Limits vary based on the organization’s edition, license, and configuration, ensuring fair use of resources.
  77. 76. What are Named Credentials, and how are they used?
    Named Credentials store authentication details (URL, credentials) for external services, making callouts secure and simplifying authentication by centralizing connection details.
  78. 77. Explain the purpose of Outbound Messaging.
    Outbound Messaging is a declarative feature that sends SOAP messages to external systems based on workflow rules or approval processes. It’s used for real-time notifications to external systems when Salesforce data changes.
  79. 78. How do you use Apex Callouts?
    Apex Callouts are used in Apex code to make HTTP requests (REST or SOAP) to external systems. This requires @future or Continuation methods for asynchronous processing when handling data-intensive responses.
  80. 79. What is Salesforce Connect, and how does it work?
    Salesforce Connect enables real-time data access from external systems without storing it in Salesforce, via External Objects. It’s useful for connecting large datasets directly, avoiding data duplication.
  81. 80. How would you handle API callouts from Flow?
    API callouts from Flow are managed using External Services or Apex Invocable Methods, which enable Flow to interact with external systems, making it more versatile for real-time integrations.
  82. 81. What is the Salesforce Data Loader, and when would you use it?
    The Salesforce Data Loader is a client application for bulk data import and export in Salesforce. It’s often used to handle large data volumes for tasks like inserting, updating, deleting, or exporting records when changes are too large for Salesforce’s native tools.
  83. 82. Describe the difference between Insert and Upsert operations.
    Insert adds new records to Salesforce, while Upsert is a combination of “update” and “insert” that either updates existing records or inserts new ones based on whether matching records are found.
  84. 83. What are Validation Rules, and how are they used?
    Validation Rules are logic-based rules used to ensure data quality by enforcing specific criteria when users enter data. If the data doesn’t meet these criteria, an error message appears, stopping the save operation.
  85. 84. Explain the purpose of Duplicate Management.
    Duplicate Management prevents and manages duplicate records using matching rules and duplicate rules. It can be configured to alert users, block duplicate entries, or allow duplicates but flag them.
  86. 85. How would you handle Data Export and Data Backup?
    Salesforce offers tools like the Data Export Service for regular, automated exports and Data Loader for one-time exports. For backups, third-party solutions or Salesforce’s backup services are used to securely store data off-platform.
  87. 86. What is a Roll-Up Summary Field?
    A Roll-Up Summary Field calculates aggregate values (like SUM, COUNT, MIN, or MAX) from related records on a parent record. It’s only available for Master-Detail relationships and is used to summarize related data.
  88. 87. How do you handle large data volumes in Salesforce?
    For large data volumes, techniques like selective SOQL queries, indexing, batch processing, archiving, and careful use of sharing rules and triggers help manage data efficiently and prevent performance issues.
  89. 88. What are Archiving and Data Retention policies?
    Archiving and Data Retention policies define how long data is kept active and when it’s archived or deleted. Salesforce automatically archives older records like activities after a set period, but custom policies may be set based on business needs.
  90. 89. What are data skew and ownership skew in Salesforce?
    Data Skew occurs when a large number of child records are linked to a single parent, causing performance issues. Ownership Skew is when a single user owns many records, which can impact sharing and visibility.
  91. 90. How would you perform data deduplication?
    Data Deduplication can be handled using tools like duplicate rules, matching rules, the Data Import Wizard, third-party tools, and manual cleanup strategies to ensure clean, unique records.

  92. 91. Explain the different deployment methods in Salesforce (Change Sets, Ant Migration, etc.).
    Change Sets allow configuration transfer between related Salesforce orgs. Ant Migration Tool and Salesforce CLI are used for more control, especially for moving metadata across environments programmatically. Salesforce DX is suited for version-controlled and complex environments.
  93. 92. What is Salesforce DX, and why is it useful?
    Salesforce DX is a development model that encourages version control, modular app development, and continuous integration. It allows developers to collaborate, test, and release changes efficiently.
  94. 93. How does the Sandbox environment work in Salesforce?
    Sandboxes are isolated environments that mirror production data and settings, allowing testing and development without affecting live data. Types include Developer, Partial Copy, and Full Copy, each serving different levels of data requirements.
  95. 94. Explain the purpose of Source Control in Salesforce development.
    Source Control tracks code changes, manages versions, and enables collaboration. By storing code in repositories (e.g., Git), teams can revert to previous versions, track changes, and enable continuous deployment practices.
  96. 95. What is a Managed Package vs. an Unmanaged Package?
    A Managed Package is a distribution of app components that can be updated and is protected, often used for commercial solutions. An Unmanaged Package is open and editable but lacks the ability to update automatically, typically used for internal distribution.
  97. 96. Describe Continuous Integration (CI) and Continuous Delivery (CD)
    in Salesforce.
    CI/CD in Salesforce automates code integration, testing, and deployment processes. CI ensures that changes are consistently tested, while CD automates moving those changes to production or staging environments.
  98. 97. How would you roll back a deployment if there is an issue?
    To roll back a deployment, restore from a previous version in source control, redeploy the last stable version, or use backup data to revert. Salesforce itself does not have an automatic rollback, so planning and backups are critical.
  99. 98. What are the various Sandbox types, and when would you use each?
    Developer Sandboxes are used for testing code/configs, Developer Pro for slightly larger data sets, Partial Copy for testing with partial production data, and Full Sandboxes for full-scale testing and training with complete production data.
  100. 99. What are Post-Deployment Steps?
    Post-Deployment Steps include actions taken after deployment, such as verifying data integrity, running tests, assigning permissions, activating flows, updating custom settings, and checking integrations to ensure everything functions as expected.
  101. 100. How does Metadata API work for deployment?
    The Metadata API is used to retrieve, deploy, create, update, or delete metadata components. It enables automated and version-controlled deployments between Salesforce environments, often via tools like Ant or Salesforce CLI.
